# Spec: Harden grouping with full plan validation and safe fallback

**Created**: 2026-06-21
**Task**: [CLO-492](https://linear.app/cloud-ai/issue/CLO-492) (slice S5)
**Estimated scope**: M (~4 source files + acceptance harness, ~5 sub-tasks)
**Extends**: CLO-487 grouping ([spec](2026-06-20-clo-487-semantic-grouping.md)) and CLO-488 typed errors + retries ([spec](2026-06-20-clo-488-typed-errors.md)). Architecture locked by [ADR-001](../adrs/001-foundational-architecture-decisions.md) Decision 6 (regenerate-per-group message contract), Decision 9 (reset a curated index with a warning; no hunk-level preservation in v1), Decision 10 (non-TTY guard).
**Covers FR**: 23 (full bijective plan validation, Must), 24 (single-commit fallback on grouping failure, Must), 46 (do not silently destroy a curated index, Should), 47 (transactional commit: generate before mutating, restore on failure/abort, Must)

---

## 1. Problem Statement

The grouping plan returned by the provider is only *partially* validated, and the project's own code says so. `plan::validate_basic` (`src/plan.rs:243`) checks four things: the plan has at least one group (`NoGroups`), group 1 is non-empty (`EmptyFirstGroup`), group 1 carries a usable message (`MissingFirstMessage`), and no group references a path absent from the real change set (`UnknownFile`). Its doc comment (`src/plan.rs:241`) and the `PlanError` enum comment (`src/plan.rs:29`) both defer the rest to this slice: *"Full bijective validation (every changed file covered exactly once) is CLO-492."* The fallback routing comment in `src/main.rs:130` likewise reads *"CLO-492 owns richer fallback policy."*

Three gaps remain, mapped to the FRs:

1. **The plan is not validated as a partition of the change set (FR-23, Must).** `validate_basic` rejects a *hallucinated* path (case a) but accepts a plan that **omits** a changed file (case b), lists a file in **more than one group** (case c), or contains an **empty group** in any position past the first (case d). An omission is the dangerous one: the omitted file is silently dropped from the grouped run and never committed, exactly the failure the PRD calls out - the bash validator (`docs/tmp/git-commit-ai.sh:382`) only builds a lookup of valid files and flags unknown suggestions; it never checks coverage, so omissions silently drop files from history (FR-23 prose). A duplicate would stage the same file under two groups across runs; an empty group is a malformed plan. FR-23 requires the plan to **partition** the change set: every changed file appears in **exactly one** non-empty group, with the commit message placed per the FR-45 contract (on `groups[0]`).

2. **Fallback policy and its "why" (FR-24, Must).** When grouping cannot produce a usable plan, the run must degrade to the single-commit flow **after** transient retries are exhausted, **warn the user that grouping failed and why**, stage **only after** the user confirms the lumped commit, and on decline **restore the pre-run index** rather than leave everything staged. The retry half is already satisfied upstream: CLO-488 wrapped the HTTP round-trip in `retry_with` inside `groq::send_chat`, so by the time `groq::generate_plan` returns an error the transient (429/5xx) retries are already spent. A *validation* failure is deterministic and is not (and must not be) re-requested - it falls back immediately. The single-commit path already stages post-confirm (`src/main.rs:242`, inside the `Decision::Commit` arm) and restores the index on a commit-step error (`src/main.rs:227-229`). The remaining work is to make the warning **explain why** grouping failed (the current `eprintln!` at `src/main.rs:96` prints the raw reason; the new validation errors must produce clear, specific reasons) and to **prove** the decline-leaves-index-untouched property end to end.

3. **A curated index is reset without a runtime warning (FR-46, Should).** Both commit paths discard the user's staging. The grouping path resets the index to HEAD (`Repo::clear_staged` -> `read-tree HEAD`, `src/git.rs:270`) and then stages exactly group 1; the single-commit path runs `Repo::stage_all` -> `git add -A` (`src/git.rs:208`). Either way a pre-existing **partial / hunk-level** staging (`git add -p`) is destroyed: v1 groups whole files, so the hunks the user deliberately *excluded* get committed. There is a **static** disclosure in `--help` (`src/cli.rs:6`, `EGRESS_DISCLOSURE`) but **no runtime warning** when a curated index is actually present. FR-46 (and ADR-001 Decision 9) require gcm to **warn before resetting** a pre-existing curated index, with particular emphasis on partial staging, which v1 does not preserve.

**Who is affected**: any gcm user whose model returns an imperfect plan (an omitted or duplicated file - silent history loss without this slice), and any user who has staged work (especially `git add -p` hunks) before running gcm. **What triggers it**: a non-partition plan from the provider; or a pre-existing staged/partial index. **Why it matters**: a commit tool must never silently drop a changed file from a grouped commit, and must never silently throw away a user's deliberate staging without warning.

**This slice does not** add a provider trait or new backend (CLO-489), touch the cache (CLO-491), broaden the retryable error set or change the taxonomy (CLO-488), or preserve hunk-level staging (an explicit v1 non-goal, ADR-001 Decision 9 - the warning is the mitigation). It hardens validation and fallback in place, in `src/plan.rs`, `src/git.rs`, `src/main.rs`, and the acceptance harness.

## 2. Acceptance Criteria

Mapped from the Linear task's three acceptance criteria, made testable:

- [ ] **AC-1 (omission -> fallback, not committed)**: A plan that **omits** at least one changed file is rejected by validation; the run does **not** produce a grouped commit and instead falls back to the single-commit flow with an announced reason naming the omitted file. Verified by an acceptance case whose mock plan covers only a subset of the changed files: the result is a single lumped commit (or a clean decline), never a group-1 commit that leaves the omitted file uncommitted.
- [ ] **AC-2 (duplicate -> fallback, not committed)**: A plan that lists the **same file in two groups** is rejected to fallback with a reason naming the duplicated file; no grouped commit is made. (Linear AC: "omits a changed file or duplicates one is rejected to fallback, not committed.")
- [ ] **AC-3 (empty group -> fallback)**: A plan containing an empty group in **any** position (not just group 1) is rejected with an `EmptyGroup` reason; falls back.
- [ ] **AC-4 (valid partition accepted)**: A plan whose groups partition the change set exactly (every changed file in exactly one non-empty group, `groups[0]` carrying a non-blank message) passes validation and commits group 1 normally - i.e. the hardened validator does **not** regress the CLO-487 happy path.
- [ ] **AC-5 (decline fallback leaves the index exactly as before)**: On a **forced grouping failure** (the provider yields no usable plan), the single-commit fallback is offered; **declining** it (answering `n`) exits 0 and leaves the index in the **same staged state** as before the run - nothing staged that was not staged before, nothing unstaged that was staged before. (Linear AC verbatim.) "Same staged state" is verified by an **identical `git write-tree` SHA** (the staged tree) **and identical `git status --porcelain=v1 -z`** before the run and after the decline - not a byte-compare of `.git/index` (which can differ in stat/timestamp metadata without any content change).
- [ ] **AC-6 (warn-why on fallback)**: Every fallback to single-commit prints a single, specific warning to stderr that (a) states grouping failed and (b) gives the reason (the `PlanError`/provider Display), before any single-commit staging. A validation-driven fallback names the offending file/group; it is never silent and never mid-stream.
- [ ] **AC-7 (pre-staged hunks warn before reset)**: Running gcm with a **pre-existing staged index** - in particular a partially-staged file (`git add -p`, status `MM`) - prints a warning to stderr **before** the index is reset/overridden, on both the grouping and `--all` real paths. **Timing (testable)**: the warning is emitted in `execute` **after** the unmerged-conflict guard (`src/main.rs:66-68`) and **before** any plan-generation, staging, or fallback warning - so on a run that also falls back, the curated-index warning precedes the "grouping failed" warning (AC-6). **Substring (testable)**: the warning text must contain the literal substrings `"curated index"`, `"reset"`, and a hunk-level-not-preserved phrase (e.g. `"hunk-level staging is not preserved"`), so acceptance tests can grep stderr for them. The warning prints even under `--yes` (it is non-blocking stderr output, never a prompt - see the Constraints decision). A `--dry-run` (which mutates nothing) prints no such warning.
- [ ] **AC-8 (fallback after retries, not before)**: A transient provider error (429/5xx) that the existing CLO-488 retry loop would retry is still retried before any fallback; a **validation** failure (non-partition plan) falls back **immediately** without re-requesting the grouping. Verified by request-count assertions on the mock (`--all` isolates one request; the grouping path gathers plan + one fallback message call, neither validation-retried).
- [ ] **AC-9 (validation is pure and unit-tested)**: The full validator is a pure function over `(&Plan, &change_set)` returning `Result<(), PlanError>`. Unit tests cover: valid partition (Ok); omission; duplicate-across-groups; empty group at index 0 and at index > 0; unknown file; no groups; missing/blank `groups[0]` message; and that each new `PlanError` variant has a distinct, non-empty `Display`.
- [ ] **AC-10 (transactional, no regressions, quality gates)**: No new crate dependency; no async. `cargo fmt --check`, `cargo clippy --all-targets -- -D warnings`, and `cargo test` are clean; `scripts/acceptance.sh` passes including the new cases. The FR-47 index transaction is preserved: on any non-commit exit (decline or pre-commit failure) the index equals the pre-run state; on a commit-step failure the group is left staged for retry (the existing `leaves_staged()` boundary is unchanged).

**Verification method**: unit tests (`cargo test`) for the pure validator (partition/duplicate/omission/empty/message-placement, `Display` distinctness) and the pure `ChangedFile` staged-state helpers; `scripts/acceptance.sh` integration cases driven by the existing stateful mock-Groq harness (`PLAN_FILE` staged with an omitting/duplicating/empty plan for AC-1/2/3; an empty `PLAN_FILE` to force the parse fallback for AC-5/6; a pre-staged `git add -p` hunk for AC-7), with before/after `git status -z` + `diff --cached` snapshots for the index-transaction assertion.

## 3. Constraints

**Must**:
- Replace `validate_basic`'s partial checks with **full partition validation** over `(&Plan, &HashSet<String> change_set)`. The plan is valid iff: (1) it has at least one group (`NoGroups`); (2) **every** group is non-empty (`EmptyGroup(index)`, generalizing the current group-1-only check); (3) `groups[0]` carries a non-null, non-blank `commit_message` (`MissingFirstMessage` - the FR-45 placement check); (4) every file named by any group exists in `change_set` (`UnknownFile`, unchanged); (5) **no file appears in more than one group** (`DuplicateFile`, new); (6) **every file in `change_set` appears in some group** (`OmittedFile`, new). Together (4)+(5)+(6) make the union of group files a bijection with `change_set`.
- Add `PlanError` variants `EmptyGroup(usize)` (0-based group index, rendered 1-based in `Display`), `DuplicateFile(String)`, and `OmittedFile(String)`, each with a distinct, specific `Display` arm (e.g. `"group {n} references no files"`, `"file '{p}' appears in more than one group"`, `"plan omitted changed file '{p}'"`). Keep `NoGroups`, `MissingFirstMessage`, `UnknownFile`, `Parse`. Remove or repurpose `EmptyFirstGroup` (generalized into `EmptyGroup(0)`); update every construction/match/test site.
- Keep the validator **pure and deterministic** (no I/O, no git calls); it takes the already-computed change set (`changed.iter().map(|c| c.path).collect()` as today, `src/main.rs:136`). Choose a **stable check order** so the reported error is deterministic for a given plan (suggested: NoGroups -> EmptyGroup(first offending) -> MissingFirstMessage -> UnknownFile(first) -> DuplicateFile(first) -> OmittedFile(first, in `changed` order)). Document the order in the function doc-comment.
- Wire the full validator into `build_plan` (`src/main.rs:137`) in place of `validate_basic`; a validation failure stays a `BuildError::Fallback(reason)` carrying the `PlanError` `Display` so the warning explains **why** (FR-24/AC-6). The existing fallback `eprintln!` (`src/main.rs:96`) already prints `"gcm: {reason}. Falling back to single-commit mode."`; keep that shape (it satisfies "warn that grouping failed and why") - the new work is ensuring `reason` is specific.
- Add a **runtime curated-index warning** (FR-46/AC-7). Add pure helpers on `ChangedFile`: `is_staged(&self) -> bool` (`x != b' ' && x != b'?'`) and `is_partially_staged(&self) -> bool` (`x != b' ' && x != b'?' && y != b' ' && y != b'?'` - the index and worktree both diverge, the `git add -p` / staged-then-modified signature). In `execute` (`src/main.rs`), after the unmerged-conflict guard (`src/main.rs:66-68`) and **before** the `--all`/grouping branch, when `!args.dry_run` and any `changed` file `is_staged()`, print one warning to **stderr** naming how many files are staged (and how many are partially staged) and stating that gcm will reset the index and re-stage by group, and that partial/hunk-level staging is not preserved in v1 (cross-reference the existing `--help` disclosure wording, `src/cli.rs:6`). `is_staged` reads the **index-side** status char (`x`, the first of the two `XY` chars). Placing the check after the `is_unmerged()` guard means an unmerged entry (whose `x` could be e.g. `U`/`A`/`D` and so satisfy `is_staged()`) never reaches the warning - conflicts abort upstream, so the warning only fires on a clean (non-conflicted) repo (Gemini note). The warning is **informational** (print and proceed); the subsequent `ui::confirm` prompt remains the user's abort gate.
- **DECISION (warning under `--yes`, flagged for the checkpoint)**: the curated-index warning **prints even under `--yes`** (and any non-interactive run). Rationale: FR-46's intent is that partial-staging loss is "documented, **not silent** as in bash" - `--yes` suppresses the confirmation *prompt*, not a data-loss *warning* on stderr, which a script can capture/log. Suppressing it under automation would reintroduce exactly the silent loss FR-46 forbids. (Reviewer split: Gemini = print/non-blocking; Ollama = consider suppressing under `--yes`. Resolved toward Gemini on FR-46 grounds; the owner may flip this at the spec checkpoint.) The warning is never a prompt and never changes the exit code.
- **Warning ordering**: when a run both has a curated index **and** falls back, the curated-index warning (emitted in `execute` before grouping) precedes the "grouping failed" fallback warning (emitted in the grouping branch). Tests assert this order on stderr.
- **Future-proofing (`--plan-only`/CLO-493)**: gate the warning on "will the index actually be mutated", not literally on `--dry-run`. Today that is `!args.dry_run`; when CLO-493 adds a no-mutation `--plan-only`, the same gate must cover it. Note this in a code comment so the gate is extended, not duplicated. `--reset` (FR-8) clears the *cache*, not the index, so a `--reset` run with a curated index **still** warns (the index is still about to be reset by grouping).
- Preserve the FR-47 index transaction exactly as built in CLO-491: `snapshot_index` before the commit attempt, `restore_index` on a non-`leaves_staged()` error, no restore on `CommitFailed` (leave the group staged for retry), no mutation on abort. Do **not** change `leaves_staged()` semantics or the `CommitOutcome` gate.

**Must-not**:
- Must not add any new crate dependency, async runtime, or logging framework (ADR-001 Decision 2). Use `std` + the already-present `serde`/`serde_json`. The warning is a plain `eprintln!`.
- Must not change the plan **schema** (`plan::plan_schema`) or the message **contract** (regenerate-per-group, ADR-001 Decision 6): `groups[0]` carries the message; later groups' messages are **null per contract but tolerated/ignored if present** (re-analysis regenerates them) - do **not** reject a plan merely because a later group has a non-null message (that would cause spurious fallbacks on benign model output). The placement check is "`groups[0]` has a usable message", nothing more.
- Must not re-request the grouping call on a validation failure, and must not retry a deterministic (non-transient) failure. Transient retries belong to CLO-488's `retry_with` and already run before `generate_plan` returns; validation failures fall back immediately.
- Must not make the curated-index warning a hard stop or an extra confirmation prompt. FR-46/ADR-001 Decision 9 say *warn*, then reset. The commit confirm is the existing gate. Do not block `--yes`/non-interactive runs on it.
- Must not attempt to preserve hunk-level staging (explicit v1 non-goal). The warning + whole-file regrouping is the agreed behavior.
- Must not warn about a curated index on a `--dry-run` (nothing is reset) or for purely **untracked** (`??`) or purely **unstaged** (` M`) files (no curation to destroy).
- Must not alter the exit-code contract (0 ok/abort, 1 runtime error, 2 usage) or the `--all`/merge bypass ordering relative to the unmerged guard (`src/main.rs:61-76`).

**Prefer**:
- Keep the validator a single free function (rename `validate_basic` -> `validate` or `validate_full`, updating the doc-comment to drop the "CLO-492" deferral note). One pass that collects the offending item per rule is fine; readability over micro-optimization (change sets are small).
- Build the per-group / cross-group file accounting with a `HashSet`/`HashMap<&str, usize>` (file -> group index) so duplicate detection is O(n) and names the first duplicate deterministically.
- Keep the FR-46 warning text short and specific; reuse the phrasing already in `EGRESS_DISCLOSURE` (`src/cli.rs:6`) so the static disclosure and the runtime warning are consistent. Consider a small `ui::warn_curated_index(staged: usize, partial: usize)` helper for testable, consistent wording.
- Reuse the existing mock-Groq acceptance harness (`PLAN_FILE`, `GCM_GROQ_BASE_URL`, the `--all` single-request path) rather than introducing a new test mechanism; mirror the existing "index restored after failed commit" assertion style (`scripts/acceptance.sh:240`).

**Escalate when**:
- The model's path representation for a file in the plan does not match `ChangedFile.path` (e.g. a quoting/normalization mismatch) such that a *correct* plan is read as an omission/unknown. FR-31 (`-c core.quotePath=false`, NUL paths) is supposed to prevent this; if validation rejects plans that are actually correct because of path encoding, that is an FR-31 regression to surface, not a validator change.
- Implementing the spec would require touching the provider abstraction (CLO-489), the cache (CLO-491), the error taxonomy/retry policy (CLO-488), or preserving hunk-level staging - that is scope creep into another slice.

**Edge cases to verify**:
- **Omission with a rename**: a `ChangedFile` rename contributes `path` (new) to `change_set`; a plan that names the new path covers it. The validator uses `path` only (the same field `change_set` is built from), so a rename is covered iff its new path is in some group - no double-counting of `orig_path` (which is a staging detail, not a change-set member).
- **Duplicate vs. legitimate**: the same path in the **same** group twice is also a duplicate (degenerate) - the file->group map flags it; assert the message still names the file.
- **All files in one group**: a single-group plan covering every changed file is a valid partition (1 group, non-empty, message present) - must pass (it is just "group everything", the natural small-change case).
- **Empty change set**: cannot occur here - `execute` returns at the no-changes guard (`src/main.rs:50`) before grouping, so `validate` is never called with an empty change_set. (Document; no test needed beyond the guard.)
