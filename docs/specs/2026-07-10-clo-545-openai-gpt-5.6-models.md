# Spec: Migrate OpenAI provider to GPT-5.6 (terra default, luna selectable), validate to the GPT-5.6 family

**Created**: 2026-07-10 · **Revised**: 2026-07-11 (owner spec-review rounds 1-2)
**Linear**: [CLO-545](https://linear.app/cloud-ai/issue/CLO-545)
**Related (out of scope, deferred)**: [CLO-547](https://linear.app/cloud-ai/issue/CLO-547) - provider-wide model-discovery hardening (capability filtering, no-inject-after-live, transport tests)
**Estimated scope**: S/M (7 files: `mod.rs`, `models.rs`, `openai.rs`, `cli.rs`, `diff.rs`, `config.rs`, `README.md`; 5 sub-tasks)

## 1. Problem Statement

gcm ships a stale OpenAI model catalog and a payload path built for the wrong model
class. The provider defaults to `gpt-5.4-mini` (`src/provider/mod.rs:376`), the
wizard's static fallback list is `["gpt-5.4-mini", "gpt-5.4", "gpt-4o-mini"]`
(`src/provider/models.rs:283`), and `src/provider/openai.rs` carries an o-series
(`o1`/`o3`/`o4`) reasoning-detection branch (`is_reasoning_model`, `system_role`,
`apply_model_params`, `apply_model_params_resolve`).

OpenAI has published the GPT-5.6 family, which supersedes these. Two models are in
scope:

- **`gpt-5.6-terra`** - mini tier, 1.05M context, 128k output, $2.50 / $15 per 1M
  (cached $0.25), cutoff 2026-02-16. **New default.**
- **`gpt-5.6-luna`** - nano tier, same context/output, $1 / $6 per 1M (cached $0.10),
  cutoff 2026-02-16. Selectable, cheaper.

**Tier mapping (owner decision, from OpenAI's upgrade guide):** the guide maps *mini →
terra, nano → luna*. The current default `gpt-5.4-mini` is a **mini** model, and gcm
uses one model for grouping, messages, **and** conflict resolution (`openai.rs:64`).
The like-for-like, quality-preserving replacement is therefore **`gpt-5.6-terra`**, not
luna. Luna stays selectable for cost-sensitive users. *(This reverses the round-1
choice of luna, which predated the tier-mapping guidance.)*

**GPT-5.6 is a reasoning family.** The OpenAI latest-model guide states that if
`reasoning_effort` is omitted, **GPT-5.6 defaults to `medium`**; `none` is a valid
level. So the payload must set `reasoning_effort: "none"` explicitly (else every
commit silently runs medium reasoning), use the **`developer`** role (reasoning models
reject `system` - see the existing `openai.rs:123` comment), and set `temperature`
(subject to the AC7 smoke - see §3 Escalate). The global `strip_think` backstop
(`mod.rs:614`) is **not** the reasoning mitigation: OpenAI reasoning tokens are billed
separately and never arrive as `<think>` in `message.content` (the only field the
extractor reads, `mod.rs:617-622`); `strip_think` stays as defensive `<think>`
compatibility only.

**Validate to the GPT-5.6 family (owner decision, Design A).** gcm accepts arbitrary
model strings - `--model` (`cli.rs:94`), `GCM_OPENAI_MODEL`/config (`resolve_model_with_source`,
`mod.rs:518`), and the wizard's broad live catalog. Applying the GPT-5.6-only payload
(`reasoning_effort:"none"`, `developer`) to a **non**-5.6 OpenAI model (e.g. `gpt-4.1`,
which is non-reasoning and rejects `reasoning_effort`) produces a broken request.
OpenAI's own upgrade guide warns: *"When old and GPT-5.6 routes share a request builder,
isolate GPT-5.6-only fields instead of applying them globally."* Since gcm officially
supports **only** the GPT-5.6 family for OpenAI, we enforce that with a **validation
gate** (not a payload-family branch): `provider::select` (`mod.rs:433`) - the single
`Result`-returning construction point used by **both** the commit path (`main.rs:397`)
and the resolve path (`resolve/mod.rs:111`) - rejects any OpenAI model that is not
`gpt-5.6-terra`/`gpt-5.6-luna` with an actionable error. The payload builders then only
ever see a supported model and stay branch-free.

**Breaking-change consequence (honest framing).** An existing user's *model selection*
is preserved by config (`apply_to_env` bridges a saved `model`→`GCM_OPENAI_MODEL`,
`config.rs:447`; `gcm config`/`--reconfigure` preserve it via `preserve_existing_models`,
`config.rs:712`). But under Design A a saved **legacy** OpenAI model (e.g.
`gpt-5.4-mini`) is now **rejected** at construction until the user re-selects a GPT-5.6
model via `gcm provider`. This is an intentional breaking change, documented in README
(AC9). Unconfigured users get `gpt-5.6-terra` as the new default.

## 2. Acceptance Criteria

- [ ] **AC1** - `gcm` with no `--model`/`GCM_OPENAI_MODEL`/saved model resolves the
  OpenAI provider to `gpt-5.6-terra`. (`ProviderId::Openai.default_model() == "gpt-5.6-terra"`.)
- [ ] **AC2** - The OpenAI static fallback list is exactly
  `["gpt-5.6-terra", "gpt-5.6-luna"]` (terra first, as the default). No `gpt-5.4*` /
  `gpt-4o*` entries remain. (The separate "static baseline appended even to a
  successful live fetch" issue is deferred to CLO-547.)
- [ ] **AC3** - The o-series detection is gone (`is_reasoning_model`, `system_role`,
  `apply_model_params`, `apply_model_params_resolve` removed) and every OpenAI request
  uses one uniform, **branch-free** payload: `"role": "developer"`,
  `"reasoning_effort": "none"`, and `"temperature"` (`0.2` plan/message,
  `ctx.temperature` resolve). Strict `json_schema` Structured Output unchanged.
  (`temperature` is subject to AC7 - see §3 Escalate/BS4.)
- [ ] **AC4** - **Validation gate**: `provider::select` rejects any OpenAI model that is
  not in the supported set `{gpt-5.6-terra, gpt-5.6-luna}` with a `ProviderError`
  (`ErrorKind::Config`) whose message names the unsupported model and points to
  `gcm provider`. This fires for **both** commit and resolve (both call `select`). A
  supported model constructs normally.
- [ ] **AC5** - No legacy OpenAI model IDs or o-series references remain **as live
  code** in `src/`. Production-code sweep is zero-match:
  `rg -n 'gpt-5\.4|gpt-4o|\bo1\b|\bo3\b|\bo4\b' src/ -g '!*test*'` (and a manual check
  that any hit inside `#[cfg(test)]` blocks is the single intentional legacy-rejection
  fixture - see AC8/BS3). The README migration note (AC9) and that one fixture are the
  **only** permitted legacy references.
- [ ] **AC6** - `README.md` documents `gpt-5.6-terra` as the OpenAI default (rows at
  `:166`, `:265`); the reasoning-model note (`:270-271`) no longer advertises o-series
  `--model` support.
- [ ] **AC7** - **Three isolated live smokes** (owner-run, needs `OPENAI_API_KEY`), each
  in an **isolated config** (temp `HOME`/config dir or empty whitelist so the pinned
  model is accepted), with a **fresh plan cache**, and **`--model` pinned** (not just
  `--provider`, since a saved model/env would otherwise win):
  1. **Grouping** - `gcm --provider=openai --model=gpt-5.6-terra --plan-only` on a
     multi-file dirty tree → a **multi-group** plan whose strict `json_schema` parses;
     debug log shows **no** "falling back to single-commit" line (exit 0 alone is
     insufficient). Exercises `build_plan_payload`.
  2. **Message** - `gcm --provider=openai --model=gpt-5.6-terra --all --dry-run` → a
     valid single commit message. Exercises `build_message_payload`.
  3. **Resolve** - on a crafted conflict,
     `gcm resolve --dry-run --conflict-auto-policy=complex --provider=openai --model=gpt-5.6-terra`
     → strict resolve schema parses, hunks are covered, no forced escalation. Exercises
     `build_resolve_payload` (its own schema + `ctx.temperature`).
- [ ] **AC8** - Every unit-test assertion carrying a legacy model string is updated to a
  GPT-5.6 id, **except** one intentional, annotated fixture that asserts the AC4
  rejection of a legacy model (e.g. a saved/overridden `gpt-5.4-mini` → `ErrorKind::Config`).
  That fixture is the AC9 regression guard and is the sole legacy string permitted in
  `src/` tests.
- [ ] **AC9** - `README.md` carries a **breaking-change migration note**: (a) OpenAI
  `--model` is now restricted to `gpt-5.6-terra`/`gpt-5.6-luna`; other OpenAI models
  (incl. legacy `gpt-5.4-mini`, o-series) are rejected with an actionable error; (b)
  the built-in default changed to `gpt-5.6-terra`; (c) existing users whose saved OpenAI
  model is legacy must run **`gcm provider`** to re-select (their selection is otherwise
  preserved but no longer valid); `gcm config`/`--reconfigure` preserve the old value
  and are **not** the migration path.

**Verification method**: `cargo test` for AC1-AC4/AC8 (default, fallback, payload shape,
validation gate, rejection fixture); production `rg` sweep for AC5; manual diff read for
AC6/AC9; owner-run CLI smokes for AC7; full gate (`fmt`/`clippy`/`test`) throughout.

## 3. Constraints

**Must**:
- `ProviderId::Openai.default_model()` → `"gpt-5.6-terra"`.
- OpenAI static fallback list → `["gpt-5.6-terra", "gpt-5.6-luna"]`.
- Introduce a single source of truth for the supported set in `openai.rs` (e.g.
  `pub(super) const SUPPORTED_MODELS: [&str; 2] = ["gpt-5.6-terra", "gpt-5.6-luna"]`) and
  use it in **both** the `select` validation gate and the OpenAI arm of
  `static_fallback_models` (and, if it doesn't complicate the match, `default_model`)
  so the list and the gate cannot drift.
- Delete `is_reasoning_model`, `system_role`, `apply_model_params`,
  `apply_model_params_resolve`; inline the uniform policy (`developer` role,
  `reasoning_effort:"none"`, `temperature`) into `build_plan_payload`,
  `build_message_payload`, `build_resolve_payload`.
- Add the `select` gate: for `ProviderId::Openai`, reject a model not in
  `SUPPORTED_MODELS` with `ProviderError { provider: "OpenAI", kind: ErrorKind::Config(msg) }`,
  `msg` naming the model and directing to `gcm provider`.
- Keep `strict: true` `json_schema` Structured Outputs unchanged.
- Update every stale model string in `src/` (incl. test fixtures) except the one
  legacy-rejection fixture (AC8). Update the `openai.rs` module doc (`:1-6`) and the
  `diff_budget` comment (`:111`) to reflect terra/luna + the 1.05M window.

**Must-not**:
- Must-not add a payload-family branch (`is_reasoning_model`-style) - the gate makes the
  builders uniform; no per-model payload variation.
- Must-not omit `reasoning_effort` (defaults to `medium`).
- Must-not rely on `strip_think` as the reasoning mitigation.
- Must-not let the validation gate and the fallback list drift (share `SUPPORTED_MODELS`).
- Must-not change the `diff_budget` numeric value; must-not add `gpt-5.6-sol`; must-not
  edit `docs/PROJECT.md` history; must-not touch shared prompts or `strip_think`.
- Must-not take on CLO-547 discovery-layer work.

**Prefer**:
- Prefer `gpt-5.6-terra` as the sample id in retargeted tests.
- Prefer deleting the dead `o_series_*` / `reasoning_model_detection` tests.

**Escalate when**:
- Any AC7 smoke shows OpenAI **rejecting `temperature`** (HTTP 400) on a GPT-5.6 model.
  This is **not a one-line change** (BS4): drop `temperature` from all three builders,
  update the three per-builder payload tests, and add a README note that
  `--conflict-temperature` (resolve) has **no effect** for OpenAI (the model ignores/
  rejects it). Record it in the workflow YAML and re-run the smokes. Keep `developer` +
  `reasoning_effort:"none"`; do **not** revive a model-family branch.
- Any AC5 residual hit is real live code (not the one fixture / README note) - surface it.

**Rollback**: constant/string edits + one deleted detection branch + one validation gate.
No schema/config-version/persisted-state migration, so reverting the CLO-545 commit(s) is
clean if a blocker survives the escalate fix.

## 4. Decomposition

1. **ST1 - Default + fallback catalog + supported-set const** - Add
   `openai::SUPPORTED_MODELS = ["gpt-5.6-terra","gpt-5.6-luna"]`. Set `default_model()`
   (`mod.rs:376`) → `gpt-5.6-terra`; set the OpenAI `static_fallback_models` arm
   (`models.rs:283`) to `SUPPORTED_MODELS`. Retarget `provider_defaults_and_tokens`
   (`mod.rs:806`); add a strict-equality assertion to
   `fallback_always_contains_default_model`
   (`== vec!["gpt-5.6-terra","gpt-5.6-luna"]`); retarget the sample ids in
   `parse_openai_compatible_data_ids` + `keep_chat_model_excludes_non_text_for_openai_groq`.
   Files: `mod.rs`, `models.rs`, `openai.rs`.
2. **ST2 - Uniform GPT-5.6 payload (tests-first, per builder)** - **Write per-builder
   exact-shape tests first** (BS5): `plan` (developer, `reasoning_effort:"none"`,
   `temperature:0.2`, strict schema), `message` (same), `resolve` (developer,
   `reasoning_effort:"none"`, `temperature == ctx.temperature`, strict schema). Then
   delete `is_reasoning_model`/`system_role`/`apply_model_params`/`apply_model_params_resolve`
   and inline the policy. Delete `o_series_*` + `reasoning_model_detection`; retarget
   remaining tests to `gpt-5.6-terra`. Update module doc + `diff_budget` comment. Files:
   `openai.rs`.
3. **ST3 - Validation gate (commit + resolve)** - In `provider::select`, after
   `resolve_model`, reject a non-`SUPPORTED_MODELS` OpenAI model with the actionable
   `ErrorKind::Config`. Unit-test the gate for a supported id (constructs) and a legacy
   id (rejects) - this is the AC8 legacy-rejection fixture and the AC9 regression guard,
   exempt from the AC5 sweep. Files: `mod.rs` (+ `openai.rs` if the helper lives there).
4. **ST4 - Sweep remaining src references** - Update `cli.rs:91` (`--model` doc example)
   and `diff.rs:36` (comment); replace the `gpt-5.4-mini`/`gpt-5.4`/`gpt-4o-mini`
   fixtures in `config.rs` (~15+ across ~L1814-2341) with `gpt-5.6-terra`/`gpt-5.6-luna`,
   preserving each test's intent (two-id whitelist fixtures → terra + luna). Keep the one
   legacy-rejection fixture (ST3). Run the AC5 production sweep. Files: `cli.rs`,
   `diff.rs`, `config.rs`.
5. **ST5 - Docs + breaking-change note + verification** - README `:166`/`:265`
   (terra default), `:270-271` (reasoning note), + the AC9 breaking-change migration
   note. Run `cargo fmt --check && cargo clippy -- -D warnings && cargo test` + the AC5
   sweep. Owner runs the three AC7 smokes. Files: `README.md`.

**Dependency order**: ST1 → ST2/ST3 (ST3's gate uses ST1's `SUPPORTED_MODELS`; ST2/ST3
are otherwise independent, both tests-first) → ST4 → ST5 last. AC7 smokes after
`cargo test` is green.

**Edge cases to verify**:
- `temperature` rejected by GPT-5.6 (HTTP 400) - Tests 10-12 trip-wire; §3 Escalate/BS4
  (full change: drop temperature in 3 builders + 3 tests + README `--conflict-temperature`
  note).
- Reasoning default confirmed `medium` if omitted (OpenAI guide) → policy sets `"none"`.
- `gpt-5.4-mini` is itself a reasoning model (default `none`) but is **not** in
  `SUPPORTED_MODELS`, so a saved/overridden `gpt-5.4-mini` is **rejected** (Test 7),
  not silently repurposed - the AC9 breaking change.
- Existing-user precedence: a saved OpenAI `model` still bridges to `GCM_OPENAI_MODEL`
  (`config.rs:447`) and is still *resolved*, but now fails the gate unless it's
  terra/luna → actionable error → `gcm provider`.
- Cache-key invalidation: `openai:gpt-5.4-mini` cached plans no longer match
  `openai:gpt-5.6-terra`; benign cache miss → regenerate.
- Wizard whitelist: an isolated-config smoke must ensure the pinned model is enabled
  (empty whitelist = unrestricted, or include terra) so `model_is_enabled` doesn't
  pre-empt the gate.
- **Out of scope (CLO-547):** the wizard can still surface incompatible live IDs and
  appends static baselines after a successful fetch.
