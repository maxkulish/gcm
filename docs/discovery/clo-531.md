## Problem framing

The primary user (Max) and future adopters hit merge/rebase/cherry-pick conflicts regularly. Today they must hand-edit every `<<<<<<<` marker, decide keep-left / keep-right / combine per hunk, validate syntax, and stage/continue manually. That is tedious for multi-file conflicts and error-prone for non-trivial code merges: it is easy to drop one branch's intent or introduce a syntax error.

We want a `gcm resolve` subcommand that turns a conflicted working tree into a reviewed, validated resolution with minimal hand-editing. The LLM should be the **last resort**, not the first mover: a layered pipeline deterministic/structural stages shrink what the model must reason about, then a focused 3-way prompt resolves only the hard hunks, and a human preview gate approves every write.

This is the right time because the foundational gcm rewrite pieces are now in place: the `Provider` trait (CLO-489), structured output + defensive parsing (CLO-487/517), config/onboarding (CLO-496/516), and privacy filtering (CLO-490/514) all exist and can be reused. Phase 1 targets only local markers, leaving remote MR/PR orchestration for a later thin wrapper.

## Existing code

### CLI dispatch (`src/cli.rs`, `src/main.rs`)

- `cli.rs` defines `Commands` enum with `Config`, `Status`, and `Provider`. Adding `Resolve` is a straightforward clap subcommand.
- `main.rs` already has subcommand-specific early returns for `Config`, `Status`, and `Provider`. A `Commands::Resolve` path can run before the normal commit flow.
- Global flags (`--json`, `--yes`, `--dry-run`, `--provider`, `--model`, `--secret-scan`) are already on `Cli` and usable by `resolve`.

### Git wrapper (`src/git.rs`)

- `Repo::is_merging()` detects `MERGE_HEAD`.
- `ChangedFile::is_unmerged()` is already used to abort the commit flow when conflicts exist.
- `Repo` already shells out with `-c core.quotePath=false` and `GIT_LITERAL_PATHSPECS=1`; adding `checkout --conflict=zdiff3 -- <paths>` and `diff --name-only --diff-filter=U` fits the existing pattern.
- No existing helper for reading file content from the working tree or for enumerating only unmerged paths, but both are thin wrappers.

### Provider layer (`src/provider/mod.rs` + backends)

- `Provider` trait: synchronous, with `generate_plan`, `generate_message`, `cache_model_id`, `diff_budget`. For conflict resolution we will likely add a new narrow method (or reuse `generate_message` with a custom prompt) that returns a structured resolution per hunk.
- `ProviderId` enum, `select()`, `pick_provider_id()`, `resolve_model()` all exist with flag > env > config > default precedence.
- Defensive parsing (`strip_think`, `parse_defensive`), retry/backoff (`http::post_json`), and error taxonomy (`ProviderError`/`ErrorKind`) are reusable.
- Backends implement structured output differently (OpenAI `response_format`, Gemini `responseSchema`, Anthropic forced tool-use, Ollama `format`). The new resolution schema must be emitted in each backend's native shape.

### Config (`src/config.rs`)

- `Config { version, default, providers }` with `ProviderConfig { id, key, endpoint, model, models }`.
- Precedence `flag > env > config > default` is implemented via `apply_to_env` only setting unset vars.
- A new `[conflict]` section can be added to `Config` while preserving forward/backward compat (unknown table deserializes to `#[serde(default)]`).

### Privacy (`src/privacy.rs`)

- `Privacy::load(repo, secret_scan)` compiles `.gcmignore` / `gcmignore` patterns and runs optional pre-send redaction/abort scanning.
- `filter_changed()` excludes ignored paths; `prepare_grouping()` / `prepare_diff()` redact secrets. These can be reused to filter conflicted files and to sanitize text sent to the provider.

### Output envelope (`src/output.rs`)

- `Envelope { v, status, mode, provider, model, plan, changed_files, ... }` with `emit()` as the only stdout writer.
- A new `ResolveReport` (or reuse `Envelope` with a new mode/status) will be added for `--json`.

### UI confirmation (`src/ui.rs`)

- `ui::confirm(message, auto_yes, quiet)` returns `Decision::Commit/Abort`; `e` opens `$EDITOR`. We can reuse this for per-file resolution prompts.

### Error taxonomy (`src/error.rs`)

- `GcmError::UnmergedConflicts` already exists. We will add `GcmError::NoConflicts` (or reuse `Git` generic) and a validation error variant.

### Baseline score: 8/10

No rewrite needed. The CLI dispatch, provider trait, config, privacy filter, confirmation UI, and JSON envelope are all in place. The work is additive: a new subcommand, a marker parser, a new provider prompt/schema, a validation gate, and a per-file write loop. The only structural choice is whether to add a new trait method for resolution or implement it as a provider-agnostic prompt wrapper.

## Approaches

### Approach A: Layered local pipeline (issue-specified)

**Summary:** `git checkout --conflict=zdiff3` â†’ optional `mergiraf` per file â†’ parse remaining hunks into `base/ours/theirs` â†’ classify trivial/moderate/complex â†’ send only complex hunks to the provider with a 3-way, function-granularity prompt â†’ validate â†’ preview â†’ write on accept.

**Pros:**
- Deterministic stages remove most hunks before any LLM call.
- Matches the existing gcm architecture (reuse `Provider`, `Privacy`, `config`, `ui`, `output`).
- Aligns with 2026 state-of-the-art reliability findings (LLMinus, Aider, JSS 2023).
- Conservative safety: preview gate + never auto-stage/continue.

**Cons:**
- More components to build (marker parser, hunk classifier, validation gate).
- `mergiraf` presence/absence requires two test paths.
- Prompt engineering for the 3-way resolution needs per-provider schema support.

**Effort:** M-L
**Risk:** low

### Approach B: Pure-LLM full-file resolution

**Summary:** For each conflicted file, send the entire file content with `zdiff3` markers to the provider and ask it to return the fully resolved file. Optionally include a short style excerpt.

**Pros:**
- Simpler implementation: one prompt + file write per conflicted file.
- No marker parser beyond reading the file; no hunk classifier.

**Cons:**
- Expensive: every hunk (even trivial keep-left/right) consumes tokens and latency.
- Higher hallucination risk: the model must reason about the whole file and may invent symbols or drop context.
- Harder to apply `.gcmignore`/secret-scan at hunk granularity.
- `mergiraf` cannot be reused to skip trivial work.

**Effort:** M
**Risk:** medium-high

### Approach C: In-process structural merge via `merge-engine`

**Summary:** Add the `merge-engine` crate as a dependency and use its program-analysis resolver as the deterministic stage instead of external `mergiraf`.

**Pros:**
- Pure Rust, no external binary dependency.
- Potentially deeper deterministic resolution than `mergiraf`.

**Cons:**
- Adds a heavy, possibly experimental dependency contrary to the issue's explicit "No new heavy Rust deps" decision.
- Requires evaluating whether `merge-engine` supports the languages/files in the user's repo.
- Does not match the established gcm pattern of shelling out to `git`/`mergiraf` on `PATH`.

**Effort:** L
**Risk:** high

## Choice: Approach A â€” Layered local pipeline

**Reason:** The issue already owner-confirmed this pipeline at discovery, and it is the only approach that scales reliability without scaling cost. It keeps the LLM as the last resort, reuses all existing gcm infrastructure, and honors the explicit constraint of no new heavy Rust dependencies. Trivial/moderate hunks are resolved deterministically or classified conservatively, and only complex hunks hit the provider. The preview gate and validation gate satisfy the safety invariants.
