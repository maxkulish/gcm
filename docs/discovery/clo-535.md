# Discovery Report: CLO-535

**Task:** Fix `gcm resolve` splice when resolution lacks a trailing newline.  
**Linear:** [CLO-535](https://linear.app/cloud-ai/issue/CLO-535/fix-gcm-resolve-splice-resolution-missing-a-trailing-newline-joins-the)  
**Branch:** `fix/clo-535-new-line`  
**PRD:** `docs/prds/clo-535-fix-resolve-splice-newline.md`

## Problem framing

Users running `gcm resolve` on a conflict whose LLM-provided resolution text omits the final newline get output where the line immediately after the conflict hunk is concatenated onto the last resolved line. The merge decision is still correct, but the file is mis-formatted (for example, a closing `}` lands on the same line as the preceding statement). The bug is provider-independent and was found during the CLO-531 post-merge smoke test. We want `reconstruct` to guarantee a clean splice regardless of whether the model remembered to end its response with a newline.

## Existing code

The splice logic lives in `src/resolve/mod.rs` in `reconstruct(file, resolutions, original)`:

- It detects the dominant line ending via `original.contains("\r\n")` (`uses_crlf`).
- For each hunk it either appends the resolved text (normalizing LF to CRLF when needed) or, in the escalated case, re-emits the original hunk lines with an explicit `\n` after each line.
- After the hunk text is pushed, the non-hunk branch appends subsequent context lines with `out.push_str(original_lines[line_no - 1]); out.push('\n');`.
- A final `if !original.ends_with('\n') && !out.is_empty() { out.pop(); }` removes the trailing newline for files that did not originally have one.

The escalated branch does not have the bug because it always emits `\n` explicitly. The resolution branch trusts the provider text to end with a newline. The section-text helper in `src/resolve/markers.rs:123` (`join_lines`) also ends its output with `\n`, but provider resolutions are not routed through it.

**Baseline score:** 8/10 — the overall architecture is sound; this is a localized normalization gap in `reconstruct`.

## Approaches identified

### Approach A: normalize trailing newline in `reconstruct`

After pushing the resolution text (including the CRLF-normalized variant), check `out.ends_with('\n')` (or `\r\n` for CRLF) and append one line ending if missing.

- **Pros:** Provider-independent, minimal diff, reuses existing line-ending detection, safe with the final `out.pop()` behavior.
- **Cons:** Adds a small amount of defensive string inspection per hunk.
- **Effort:** S
- **Risk:** low

### Approach B: enforce trailing newline at the provider/schema layer

Update the resolution prompt or response schema to require that resolution text end with a newline.

- **Pros:** Moves the invariant closer to the source.
- **Cons:** Not reliable across providers (some models may ignore formatting instructions), requires prompt changes and re-validation of multiple providers, does not protect against future providers.
- **Effort:** M
- **Risk:** medium

### Approach C: always trim the resolution and append a fresh line ending

Strip any trailing whitespace/line endings from the resolution and then append the correct line ending based on `uses_crlf`.

- **Pros:** Very explicit; avoids double newlines.
- **Cons:** Slightly more aggressive than necessary; could mask legitimate trailing spaces intentionally preserved by the model.
- **Effort:** S
- **Risk:** low

## Chosen approach

**Approach A: normalize trailing newline in `reconstruct`.**

Reason: it is the smallest, most defensive change. It fixes the bug for every provider without changing prompts or schemas, preserves CRLF handling, and does not interfere with the existing final-newline preservation logic.
