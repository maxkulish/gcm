# Discovery Report: CLO-494 — Add Anthropic provider via forced tool-use

**Linear:** [CLO-494](https://linear.app/cloud-ai/issue/CLO-494/add-anthropic-provider-via-forced-tool-use)
**Branch:** `feat/clo-494-anthropic`
**Date:** 2026-06-22

## Problem framing

The primary user (Max) aliases bare `gcm` to Anthropic Haiku via the subscription
`claude` CLI in the bash tool. The Rust rewrite must call the Anthropic Messages API
directly (ADR-001 Decision 3 — no `claude` CLI in the runtime), and Anthropic has no
generic `response_format` like Groq/OpenAI/Ollama. Structured output must instead be
obtained via forced tool-use: define a tool whose `input_schema` is the Plan schema,
set `tool_choice: {type:"tool"}` to force the call, and extract the typed plan from
the `tool_use` content block. This is the fourth provider behind the CLO-489 trait,
extending the capability matrix (FR-13) and closing the personal-cutover path.

### Key integration points

1. **`ProviderId` enum** — add `Anthropic` variant
2. **`select()` function** — add `ProviderId::Anthropic => Box::new(anthropic::Anthropic::new(model))`
3. **`default_model()` / `model_env_vars()`** — add Anthropic entries
4. **`HttpRequest` struct** — Anthropic requires `anthropic-version` header in addition
   to `x-api-key`; current struct only carries one auth header → needs extension
5. **`cli.rs`** — update help text to list `anthropic` as a provider option
6. **`pick_provider_id()` error message** — add "anthropic" to the valid names list

### Baseline score: 8/10

No rewrite needed. The trait, shared HTTP transport, error taxonomy, plan schema,
defensive parser, and `dimd` strip are all in place. The Gemini backend demonstrates
exactly how a divergent API shape is handled. The only structural change is extending
`HttpRequest` to support an extra header (`anthropic-version`).

## Approaches

### Approach A: Forced tool-use (as the issue title specifies)

**Summary:** Define a tool with `input_schema` = Plan schema; force it with
`tool_choice: {type:"tool", name:"commit_plan"}`; extract JSON from the `tool_use`
content block's `input` field.

**Request shape (plan):**
```json
{
  "model": "claude-haiku-4-5",
  "max_tokens": 4096,
  "system": [{"type":"text","text":"<GROUPING_SYSTEM_PROMPT>"}],
  "messages": [{"role":"user","content":"<grouping_user_content>"}],
  "tools": [{
    "name": "commit_plan",
    "description": "Return the commit grouping plan",
    "input_schema": { "type":"object", "properties":{...}, "required":[...] }
  }],
  "tool_choice": { "type":"tool", "name":"commit_plan" }
}
```

**Response shape:**
```json
{
  "content": [
    { "type":"tool_use", "name":"commit_plan", "input":{"groups":[...]} }
  ],
  "stop_reason":"tool_use"
}
```

**Pros:**
- Works on **all** Claude models (tool-use is universally available, not gated to
  specific model versions)
- The issue title explicitly says "forced tool-use"
- `input_schema` uses standard JSON Schema (same as OpenAI strict mode) — can reuse
  `plan::schema()` directly
- CoT/reasoning is separated into `thinking` content blocks (not mixed into the tool
  input), so reasoning never reaches the plan

**Cons:**
- More complex request/response than a plain text call
- `tool_use` block extraction is a new response parser (not `extract_openai_content`)
- Adds ~290-410 tool-use system prompt tokens (per Anthropic docs)

**Effort:** M
**Risk:** low

### Approach B: JSON outputs via `output_config.format`

**Summary:** Use the newer `output_config: {format: {type:"json_schema", schema:{...}}}`
parameter; response is JSON text in `content[0].text`.

**Request shape (plan):**
```json
{
  "model": "claude-haiku-4-5",
  "max_tokens": 4096,
  "system": [{"type":"text","text":"<GROUPING_SYSTEM_PROMPT>"}],
  "messages": [{"role":"user","content":"<grouping_user_content>"}],
  "output_config": {
    "format": {
      "type":"json_schema",
      "schema": { "type":"object", "properties":{...}, "required":[...] }
    }
  }
}
```

**Response shape:**
```json
{
  "content": [{ "type":"text", "text":"{\"groups\":[...]}" }],
  "stop_reason":"end_turn"
}
```

**Pros:**
- Simpler request (no `tools`/`tool_choice` ceremony)
- Response is plain JSON text — can use a simpler extractor
- Grammar-constrained sampling (like OpenAI strict mode) for guaranteed schema compliance

**Cons:**
- Only available on **Opus 4.8 / Sonnet 4.6 / Haiku 4.5** — older models (3.5, 4.0)
  don't support it, limiting model override flexibility
- The issue title specifies "forced tool-use" — deviating requires explicit justification
- `output_config.format` is newer and may have undocumented constraints
- Schema subset is more restrictive than tool-use `input_schema` (no `$ref`, `oneOf`, etc.)

**Effort:** S-M
**Risk:** medium (model-version gating)

## Choice: Approach A — Forced tool-use

**Reason:** The issue title explicitly specifies forced tool-use. Tool-use is
universally available across all Claude models (not gated to specific versions like
`output_config.format`), so users can override `--model` to any Claude model without
risking a 400. The `input_schema` accepts standard JSON Schema, so `plan::schema()`
can be reused directly. Reasoning/thinking is cleanly separated into `thinking`
content blocks, satisfying the "no reasoning in the commit message" AC. The only
structural change is extending `HttpRequest` to carry the `anthropic-version` header.
