# Discovery Report: CLO-496 — First-Run Onboarding Wizard

**Date**: 2026-06-22
**Task**: [CLO-496](https://linear.app/cloud-ai/issue/CLO-496)
**Branch**: `feat/clo-496-onboarding-wizard`

---

## Problem Framing

End users running `gcm` for the first time are affected. Today `gcm` expects the user to manually export provider API keys (`GROQ_API_KEY`, `GEMINI_API_KEY`, `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`) and optionally `GCM_PROVIDER` before the first run. There is no persistent config file, no guided setup, and no `--reconfigure`. A missing key produces a runtime `ProviderError::MissingKey`, and the `--help` text is the only documentation for the required env vars.

We want to detect a missing or empty configuration and launch an interactive wizard that walks the user through enabling providers, capturing keys or a local endpoint, choosing a default, and persisting the result. In a non-TTY context the tool must print the needed config/env and exit non-zero, preserving unattended/CI safety.

This is the right time because the prerequisites — config format decisions (CLO-485) and the provider registry (CLO-489) — are both complete.

## Existing Code

- `src/main.rs` — entry point `run()` calls `provider::select()`; missing-key errors are already surfaced as `GcmError::Provider`.
- `src/provider/mod.rs` — provider selection precedence is flag > env > default(groq); `ProviderId` enum lists Groq/Google/OpenAI/Anthropic/Ollama. Each backend reads its own key lazily from env.
- `src/cli.rs` — `Cli` struct currently exposes `--provider`, `--model`, and a small set of run flags. No config-related flags exist.
- `src/error.rs` — `GcmError::NonInteractive` already handles the "needs terminal but absent" case; we can reuse it or add a dedicated onboarding error.
- `Cargo.toml` — currently depends on `clap`, `ureq`, `serde`, `serde_json`, `tempfile`, `directories`, `sha2`. No dialog/prompt crate is present.

No config module exists. Provider defaults and model env vars live in `provider::ProviderId`. The existing CLI contract is narrow and stable; onboarding must be additive only.

**Baseline score**: 7/10. The provider registry and env-based selection are solid. The gap is entirely the absence of config persistence and an onboarding path, so no rewrite is needed — we add a `config` module and surface it through CLI.

## Approaches Identified

### 1. TOML config file in OS config dir with interactive prompts

**Summary**: Add `src/config.rs` that reads/writes `~/.config/gcm/config.toml` (or OS equivalent via `directories`). Add `--reconfigure` / `gcm config` subcommand that runs an interactive wizard using plain `std::io` prompts (no extra crate). Reuse env vars when present, else prompt with hidden input.

**Pros**:
- No new external dependencies; fits existing sync, no-async ethos.
- TOML is human-readable and easy to edit by hand if the wizard is skipped.
- `directories` crate already in `Cargo.toml`.

**Cons**:
- Plain `std::io` prompts are basic (no arrow keys, no validation loops with nice UX).
- Need to design and document the config schema ourselves.

**Effort**: M
**Risk**: low

### 2. Dialoguer/inquire crate + JSON config file

**Summary**: Use a dedicated prompt crate such as `dialoguer` or `inquire` for checkbox/multi-select UX and store config as JSON.

**Pros**:
- Better UX for selecting multiple providers and a default.
- JSON maps directly to a `serde` struct.

**Cons**:
- Adds an external dependency purely for UX polish.
- JSON is less hand-editable than TOML for this team.
- Extra dependency increases supply-chain and build risk for a single feature.

**Effort**: M
**Risk**: medium

### 3. Env-only onboarding (no persistence)

**Summary**: Instead of writing a file, generate a shell snippet (`export GROQ_API_KEY=...`) for the user to paste into their profile.

**Pros**:
- Zero persistence complexity and zero file-permission concerns.
- Minimal change to the codebase.

**Cons**:
- Does not satisfy the acceptance criterion to persist config and choose a default provider.
- Requires manual profile editing on every setup.

**Effort**: S
**Risk**: low

## Approach Chosen

**Approach 1**: TOML config in OS config dir with interactive `std::io` prompts.

Reason: it satisfies all acceptance criteria without adding a new dependency, matches the existing Rust/sync/no-async architecture, and leverages the `directories` crate already in use. The UX can be perfectly acceptable with numbered menus and hidden stdin reads for secrets.

## Discovery Debt

- Final exact TOML schema for provider entries (pending design doc).
- Whether to validate keys via a lightweight ping (out of scope per PRD, but may resurface in review).
- How `gcm config` integrates with the existing top-level `gcm` command vs a dedicated subcommand path (Clap subcommands require design).

All debt items will be resolved in the design phase; no blockers for proceeding.
