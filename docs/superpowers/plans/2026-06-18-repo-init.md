# /repo-init Command Suite Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build a user-level `/repo-init:*` Claude Code command suite that bootstraps a fresh GitHub repo (or retrofits an existing one) with a language scaffold plus the `.claude`/`.pi`/`.lok`/`docs` agent tooling migrated from a reference repo.

**Architecture:** One orchestrator command (`/repo-init:orchestrate`) runs preflight + a single up-front Q&A, writes a `.repo-init/config.yaml` manifest, then fans out independent stage commands (`scaffold`, `claude`, `pi`, `docs`) as parallel subagents over disjoint output paths, and finishes with a `verify` stage. Each stage is a standalone subcommand and the single source of truth for its logic. Language scaffolds come from `.tmpl` template sets (Golang authored; Rust+TS/Python stubbed); tooling/docs come from a reference repo (default `lok`) with project-specific values transformed out.

**Tech Stack:** Markdown Claude Code commands, `.tmpl` template files, bash, `gh`, `git`, `go`, `rg`, `npm` (for `.pi` TS extensions).

## Global Constraints

- **Install path:** `~/.claude/commands/repo-init/` (symlink into the `~/Work/investigations` git repo at `macos/configs/claude/commands/repo-init/`).
- **Entry point:** `/repo-init:orchestrate`. NO bare top-level `repo-init.md` (unsupported coexistence with the same-named dir).
- **Only `.md` files register as commands.** Every template/support file MUST use a non-`.md` suffix (`.tmpl`). Stubs are `STUB.tmpl`, never `STUB.md`.
- **Nested namespacing:** `repo-init/<x>.md` → `/repo-init:<x>`.
- **Command files:** add YAML frontmatter with a `description:`; keep each file under ~300 lines.
- **Commit targets (two repos):**
  - Command-suite files (`repo-init/**`): commit in the `investigations` repo, branch `feat/repo-init`, via a subshell: `( cd "$HOME/Work/investigations" && git add macos/configs/claude/commands/repo-init && git commit -m "..." )`.
  - This plan/spec and the fresh-mode end-to-end test output: commit in the `gcm` repo.
- **Commit style:** Conventional Commits. NEVER add "Generated with Claude Code" or "Co-Authored-By: Claude". Use regular dashes, never em dashes.
- **Reference repo default:** `/Users/mk/Code/orchestrator/lok`.
- **Manifest schema** (`.repo-init/config.yaml`, written by the orchestrator, read by every stage):

```yaml

module_path: github.com/maxkulish/gcm
owner: maxkulish
repo: gcm
stack: golang                  # golang | rust-ts | python
mode: fresh                    # fresh | retrofit
reference_repo: /Users/mk/Code/orchestrator/lok
vcs: github
linear: null                   # or {workspace: <ws>, prefix: <PFX>}
groups: all                    # all | [group, ...]
docs_convention: designs       # designs | design-docs
overwrite_policy: ask          # ask | overwrite | skip | backup

```

- **Value-transform map** (applied by `claude`, `pi`, `docs` stages to migrated text):

| From (in `lok` reference) | To | Skip when |
|---------------------------|----|-----------|
| `lokomotiv`, `lok` (word-boundary) | `{{tool_name}}` | never |
| `CLO-`, `clo-` (word-boundary) | `{{PREFIX}}-`, `{{prefix}}-` | `linear == null` |
| `linear.app/cloud-ai`, `cloud-ai` | `linear.app/{{workspace}}`, `{{workspace}}` | `linear == null` |
| `docs/design-docs/` ↔ `docs/designs/` | the path named by `docs_convention` | never |
| `docs/prds/` ↔ `docs/prd/` | `docs/prds/` (canonical) | never |

When `linear == null`: drop Linear MCP permissions from `settings.json` and leave issue-prefix references as literal `TASK-` placeholders.

# /repo-init:orchestrate - bootstrap a repo

Run from inside a freshly-cloned (or existing) repo. `$ARGUMENTS` may include `--dry-run`.

## Phase 0 - Preflight

1. Assert git repo: `git rev-parse --show-toplevel` (abort if not).
2. Detect mode: if any of `go.mod`, `Cargo.toml`, `pyproject.toml`, `package.json`, or a non-empty `src/` exist → `retrofit`; else `fresh`. Print the detected mode and confirm with AskUserQuestion (Fresh / Retrofit).
3. Detect VCS + identity: `git remote get-url origin`; parse `github.com[:/ ]<owner>/<repo>`. If no GitHub remote, ask for `owner`/`repo`. Derive `module_path = github.com/<owner>/<repo>`, `tool_name = <repo>`.
4. In retrofit mode, auto-detect `stack`: `Cargo.toml`→rust-ts, `go.mod`→golang, `pyproject.toml`→python.

## Phase 1 - Single Q&A pass

Ask (AskUserQuestion), defaulting from detected values:
1. Stack (fresh only): `Golang` | `Rust + TypeScript` | `Python`.
2. Reference repo: default `/Users/mk/Code/orchestrator/lok`, or a custom path. Validate it has `.claude/` (warn if no `.pi/`/`.lok/`/`docs/`).
3. Confirm `tool_name` and `module_path`.
4. Linear: `None` (default) | provide `workspace` + `prefix`.
5. Command/extension groups: `All` (default) or a subset.
6. `docs_convention`: `designs` (default) | `design-docs`.
7. `overwrite_policy`: `ask` (default) | `overwrite` | `skip` | `backup`.
8. `go_version` (golang only): default `1.26`.

## Phase 2 - Manifest

Write `$PWD/.repo-init/config.yaml` with all answers (schema: see the repo-init design spec).
In retrofit mode, ensure `.repo-init/` is in `.gitignore` (append if missing; in fresh mode the scaffold's `.gitignore` already includes it).
If `--dry-run`: print the manifest and the list of stages that WOULD run, then exit.

## Phase 3 - Fan out (parallel subagents)

Dispatch one subagent per ACTIVE stage in a single message (disjoint output paths):
- fresh mode: `scaffold`, `claude`, `pi`, `docs`
- retrofit mode: `claude`, `pi`, `docs` (no `scaffold`)

Each subagent prompt: "Set working directory to `<$PWD>`. Read and execute the instructions in `~/.claude/commands/repo-init/<stage>.md` using `$PWD/.repo-init/config.yaml`. Write only to that stage's owned paths. Report what you wrote." Wait for all to finish (barrier). If parallel dispatch is unavailable, run the stages sequentially.

## Phase 4 - Verify + report

1. Execute `~/.claude/commands/repo-init/verify.md` in the main thread.
2. Print a final summary: mode, stack, files created (tree), verify result, any stage failures.
3. If a stage failed: report which, keep the others, offer to re-run just that stage (`/repo-init:<stage>`).
4. Offer the initial commit: `git add -A && git commit -m "chore: bootstrap repo via /repo-init"`.

## Notes

- All interaction happens in Phase 1; stages are non-interactive.
- Never write outside `$PWD` except the manifest (under `$PWD/.repo-init/`).
